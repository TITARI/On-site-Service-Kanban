# wxauto 完整值守接入设计

## 1. 背景与目标

本阶段把 Windows 微信客户端通过 wxauto 接入内部协同看板，形成“完整值守”能力：监听所有微信新消息，由系统过滤普通聊天，自动推进用户注册绑定、缺失信息追问、工单建单、催单、处理组通知和处理回执。

当前项目已有基础能力：

1. 入站接口：`POST /api/integrations/wechat/messages`。
2. 消息分析服务：`message-intake-service` 可识别展位号、问题类型、建单或催单意图。
3. wxauto REST 桥接脚本：`scripts/wxauto-rest-bridge.mjs` 已能从 wxauto REST 服务拉取消息并转发到系统。
4. 管理页已有微信/企微 MCP 配置和近期消息展示。

本设计的目标是在现有基础上补齐可长期值守的业务闭环，而不是把业务判断写进桥接脚本。

## 2. 范围

本阶段包含：

1. wxauto 监听所有微信新消息。
2. 系统侧过滤普通聊天，只处理注册绑定、现场诉求和追问回复。
3. 微信用户自动注册系统用户，并绑定微信身份。
4. 缺失身份信息或工单字段时自动追问。
5. 自动建单、催单、处理组通知、反馈人回执。
6. 出站通知队列与发送状态记录。
7. 管理页展示微信身份、系统用户、追问会话、出站通知失败项。

本阶段不包含：

1. 微信客户端 UI 自动点击以外的非 wxauto 通道实现。
2. 群成员身份强校验，例如企业通讯录认证。
3. 多台微信机器人并发抢占同一会话。
4. 长期数据库迁移；仍沿用当前文件存储，数据模型预留后续迁移边界。

## 3. 推荐架构

采用“wxauto REST 桥 + 系统值守引擎”。

wxauto 负责收发微信消息。桥接脚本只做消息标准化、去重、转发和发送出站通知，不判断业务。

系统负责业务判断。入站消息进入系统后，由值守引擎按固定顺序处理：

1. 记录原始入站消息。
2. 创建或更新微信身份 `ChatIdentity` 与会话 `Conversation`。
3. 判断是否为注册绑定消息。
4. 推进已有追问会话。
5. 过滤普通聊天。
6. 分析现场诉求。
7. 创建工单、催单或进入追问。
8. 写入出站通知队列。

这样后续接企业微信、公众号或其他消息通道时，工单、绑定、追问和通知逻辑都可以复用。

## 4. 消息流

### 4.1 入站

wxauto REST 桥监听所有新消息，不设置白名单。桥接字段统一为：

1. `channel`: 固定 `wechat`。
2. `externalMessageId`: wxauto 原始消息 ID，缺失时由会话、发送人、内容、时间生成指纹。
3. `senderId`: 微信外部用户标识，能取到时必须保留。
4. `senderName`: 微信昵称或会话内显示名。
5. `senderGroup`: 群聊名或会话名。
6. `sourceConversationId`: 微信会话标识，能取到时必须保留。
7. `text`: 文本内容。
8. `imageUrls`: 图片路径或 URL。
9. `receivedAt`: 消息时间。
10. `raw`: 可选原始摘要，便于排查字段映射问题。

系统根据 `externalMessageId` 去重。重复消息不再次触发建单、追问或出站通知。

### 4.2 出站

系统不直接调用 wxauto 发送消息，而是写入 `OutboundMessage` 队列。桥接脚本轮询待发送通知，调用 wxauto REST 发送到指定个人或群聊。

出站通知记录：

1. 目标会话。
2. 目标微信用户或群。
3. 文本内容。
4. 关联工单或追问会话。
5. 状态：`pending`、`sending`、`sent`、`failed`。
6. 重试次数。
7. 最近错误。
8. 创建、更新时间。

桥接脚本发送成功后回调系统标记为 `sent`；失败时写入错误并按重试策略继续。

## 5. 注册绑定

### 5.1 数据模型

新增或补齐三类记录：

1. `Person`: 系统用户，包含真实姓名、手机号、身份组、角色、启用状态。
2. `ChatIdentity`: 微信身份，包含平台、微信外部用户 ID、展示名、绑定的 `personId`、首次和最近活跃时间。
3. `Conversation`: 微信会话，包含个人或群聊标识、标题、关联人员、默认通知开关。

现有 `types.ts` 已有这些类型雏形，实施时优先复用并补齐存储字段。

### 5.2 主动注册

用户可在微信发送：

```text
注册 身份组 真实姓名 手机号
```

示例：

```text
注册 搭建组 张三 13800138000
```

系统校验：

1. 身份组必须存在且启用。
2. 真实姓名不能为空。
3. 手机号必须符合中国大陆手机号基本格式。
4. 当前微信身份可被绑定或重新绑定。

校验通过后，系统自动创建或更新 `Person`，绑定当前 `ChatIdentity`，并立即生效。

### 5.3 自动追问注册

陌生微信用户发来现场诉求时，系统先追问身份信息：

```text
请补充身份组、真实姓名、手机号，例如：注册 搭建组 张三 13800138000
```

如果用户只补充部分字段，系统只继续追问缺失项。信息补齐后，系统自动注册并继续处理最初那条现场诉求，不要求用户重复描述。

### 5.4 冲突处理

手机号已存在时：

1. 同名：更新该 `Person` 的微信绑定并立即生效。
2. 不同名：以手机号为主，更新最近微信身份记录；管理页标记为“姓名不一致”供管理员后续修正，但不阻断注册。
3. 身份组变更：立即更新 `Person.groupName`，保留时间线或审计备注。

该策略符合用户要求：自动注册并立即生效。

## 6. 自动追问

追问会话使用 `PendingWorkOrderSession` 表达。每个会话绑定微信身份和原始消息。

缺失字段包括：

1. `identityGroup`: 身份组。
2. `name`: 真实姓名。
3. `phone`: 电话号。
4. `boothNumber`: 展位号。
5. `issueType`: 问题类型。

追问规则：

1. 未绑定身份时，优先追问身份组、真实姓名、手机号。
2. 已绑定但缺少展位号时，追问展位号。
3. 有展位号但问题类型不明确时，追问问题类型，并给出可选类型。
4. 用户补齐字段后，系统复用原始诉求文本和图片继续建单或催单。
5. 一个微信身份同一时间只保留一个活跃追问会话，新的明确诉求可替换旧草稿。

追问避免打扰：

1. 普通聊天只记录不回复。
2. 明显不是现场诉求的图片或表情不追问。
3. 同一个缺失项短时间重复出现时不重复刷屏。

## 7. 过滤规则

系统先用规则过滤，再用现有消息分析兜底。

优先处理：

1. 以 `注册`、`绑定` 开头的消息。
2. 活跃追问会话的回复。
3. 包含展位号、报修、故障、处理、催、加急、无法、不能、断网、没电、桌椅、搭建等现场关键词的消息。
4. 带图片且文字含现场诉求的消息。

静默记录：

1. 问候、闲聊、确认收到等普通聊天。
2. 无文字且无法识别用途的表情或图片。
3. 系统自己发出的消息，避免回环。

不确定时：

1. 置信度较低但有现场关键词，进入人工复核或轻量追问。
2. 完全无法判断时静默，避免机器人打扰群聊。

## 8. 工单联动

消息进入工单链路后，复用 `message-intake-service` 的现有能力：

1. 自动识别展位号。
2. 自动推断问题类型。
3. 同展位同问题类型存在未关闭工单时催单。
4. 能识别完整信息时自动创建工单。
5. 不完整时进入追问会话。

建单成功后给反馈人回复：

```text
已创建工单：{工单标题}
展位：{展位号}
类型：{问题类型}
当前状态：待受理
```

催单成功后给反馈人回复：

```text
已关联已有工单并催单：{工单标题}
当前催单次数：{次数}
```

处理组通知：

1. 新工单自动派发后，通知对应处理组群或负责人。
2. 催单升级时通知处理组。
3. 超时分析产生建议时通知处理人和管理员。

处理完成和关闭回执：

1. 工单进入 `已解决` 时通知所有反馈人。
2. 工单进入 `已关闭` 时发送关闭回执。
3. 处理回复可按配置同步到微信原会话。

## 9. 管理配置

管理页需要补齐以下配置和视图：

1. wxauto REST 地址、token、轮询间隔、发送重试上限。
2. 微信接入启用状态和密钥环境变量。
3. 身份组列表，沿用现有用户分组。
4. 微信身份列表：已绑定、未绑定、姓名冲突。
5. 追问会话列表：缺失字段、原始诉求、最近追问时间。
6. 出站通知列表：待发送、发送失败、最近错误。
7. 每个身份组的默认通知目标会话，例如搭建组微信群。

敏感信息如 token 和密钥仍通过环境变量配置，不写入 `data/app-state.json`。

## 10. 桥接脚本职责

`scripts/wxauto-rest-bridge.mjs` 扩展为双向桥：

1. 初始化 wxauto。
2. 轮询所有新消息。
3. 标准化消息并推送到系统入站接口。
4. 轮询系统待发送出站通知。
5. 调用 wxauto REST 发送文本消息。
6. 回调系统更新发送状态。
7. 维护内存去重窗口，系统侧再做持久去重。
8. dry-run 模式可打印入站和出站，不写入系统或不实际发送。

桥接脚本不保存业务状态。进程重启后的可靠性由系统侧消息去重和出站通知状态保证。

## 11. 错误处理

1. wxauto 初始化失败：桥接脚本持续重试并写日志。
2. 入站接口失败：桥接脚本重试，系统侧通过 `externalMessageId` 防止重复建单。
3. 出站发送失败：通知保留 `failed` 状态和错误原因，达到重试上限后等待人工处理。
4. 注册字段错误：回复明确纠错信息。
5. 身份组不存在：回复可选身份组。
6. 展位号不存在主数据：允许建单，标记未知公司，并通知管理员补主数据。
7. AI 或分类失败：进入人工复核或追问，不阻断消息记录。

## 12. 测试策略

优先按 TDD 实施。

服务层测试：

1. 主动注册消息能创建 `Person` 并绑定 `ChatIdentity`。
2. 陌生用户现场诉求会创建追问会话。
3. 追问补齐身份后继续处理原始诉求。
4. 缺展位号时追问展位号。
5. 同展位同类型消息触发催单。
6. 普通聊天静默记录，不写出站通知。
7. 工单状态变更写入反馈人回执。

API 测试：

1. 入站接口接受 wxauto 字段别名。
2. 入站接口鉴权失败不写状态。
3. 出站通知拉取接口只返回待发送项。
4. 发送结果回调能更新状态和错误。

桥接脚本测试：

1. 字段映射覆盖常见 wxauto 消息结构。
2. 图片路径归一化。
3. dry-run 不调用真实发送。
4. 发送失败时保留错误并继续轮询。

构建验证：

1. `npm run test:run`
2. `npm run build`
3. 本地启动后用 dry-run 验证入站、追问、出站日志。

## 13. 验收标准

1. 微信任意会话新消息会进入系统记录。
2. 普通聊天不会触发自动回复。
3. 陌生用户发送现场诉求时，系统自动追问身份组、真实姓名、手机号。
4. 用户发送 `注册 身份组 真实姓名 手机号` 后自动注册并立即生效。
5. 缺展位号或问题类型时自动追问。
6. 信息补齐后自动建单或催单。
7. 建单、催单、解决、关闭会回发微信。
8. 处理组能收到新工单和催单升级通知。
9. 出站通知失败可在管理页看到，并保留错误原因。
10. 桥接脚本重启不会导致同一微信消息重复建单。
