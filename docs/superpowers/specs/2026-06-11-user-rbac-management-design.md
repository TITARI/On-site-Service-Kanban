# 用户与权限管理 RBAC 设计

## 背景

当前系统已经具备独立 PC 后台、用户分组配置、`people` 人员表、`chat_identities` 微信/企微身份表、审计日志和导入任务表，但尚未形成完整的账户与权限体系：

- 移动端用户通过姓名、手机号和分组免密进入，身份与权限保存在浏览器本地。
- 移动端可以自行选择用户分组，服务端写操作仍接收客户端传入的操作者信息。
- 后台通过统一口令登录，无法识别具体管理员，也无法按用户撤销后台权限。
- 后台只能查看人员和身份数量，不能维护人员资料、账号状态或微信绑定。
- 微信消息注册可以创建人员并绑定身份，但缺少统一的后台管理和冲突处理入口。
- 展览数据支持表格导入，人员数据尚无预览、冲突决策和批量提交能力。

本次功能将建立完整 RBAC 账户体系，并在后台新增“用户与权限”模块，用于管理所有用户的资料、分组、权限继承、微信/企微绑定、后台密码和启用状态，同时提供带冲突预览的批量导入。

## 目标

- 建立服务端账户、角色、权限、凭据和会话模型。
- 用户权限统一由所属分组对应的角色继承。
- 保留移动端免密登录和用户自选分组的使用方式。
- 移动端登录时按手机号自动建档或更新人员信息。
- 管理员可以锁定用户分组；锁定后移动端登录不能覆盖后台分组。
- 停用用户后，其现有会话和后续请求立即失效。
- 后台改为管理员用户使用手机号和独立密码登录。
- 后台支持人员增删改查、停用、密码重置和微信/企微身份管理。
- 支持通过表格预览、处理冲突后批量新增或覆盖用户。
- 所有敏感操作写入审计日志。

## 非目标

- 第一版不开放自定义权限项或任意权限表达式。
- 第一版不允许给单个用户覆盖分组权限。
- 第一版不引入短信验证码、微信扫码登录或找回密码服务。
- 第一版不通过导入文件设置后台密码。
- 第一版不改变现有四项业务权限的含义。
- 第一版不重做工单状态流、微信自动建单或展览主数据模型。

## 已确认的业务规则

### 权限继承

权限只由用户所属分组继承，不提供用户级权限覆盖。第一版固定四项权限：

- `ticket.claim`：认领工单。
- `ticket.process`：处理工单。
- `ticket.accept`：验收工单。
- `admin.access`：进入后台并执行后台管理操作。

现有用户分组配置中的“认领、处理、验收、后台管理”映射到上述权限。界面继续以用户分组为主要配置入口，底层通过 RBAC 角色与权限关系持久化。

### 移动端免密登录

移动端继续允许用户填写姓名、手机号并选择任意启用分组：

- 手机号不存在时，创建人员和账户。
- 手机号已存在且分组未锁定时，更新姓名和所选分组。
- 手机号已存在且分组已锁定时，保留后台设置的分组，忽略移动端选择。
- 已停用用户禁止登录。
- 登录成功后创建服务端会话，浏览器不再自行决定用户权限。

该规则保留现场快速使用能力。系统将通过审计日志记录移动端导致的姓名或分组变更，使管理员可以追溯。

### 微信与企微身份

- 每个用户在微信和企微平台各最多绑定一个账号。
- 每个平台的外部账号标识全局唯一，不能同时属于多个用户。
- 后台既可以从系统已识别的身份中选择，也可以手工填写稳定的外部账号标识。
- 手工填写时创建正式身份记录，不创建临时身份。
- 绑定已被其他用户占用的身份时，必须明确确认换绑。
- 换绑、解绑和手工绑定都写入审计日志。

### 删除与停用

- 无历史关联的用户可以物理删除。
- 存在工单、消息、会话、已使用的微信身份或以该用户为操作者的审计记录等业务历史时，只能停用。
- 仅有用户维护审计、尚未提交的导入预览或从未参与消息收发的手工身份绑定，不视为业务历史；删除时保留脱敏审计快照，并清理这些从属记录。
- 停用会撤销该用户所有活动会话。
- 最后一位启用且拥有 `admin.access` 权限的管理员不能被停用、删除、移出管理员分组或撤销后台权限。

## 数据模型

### 既有人员表

`people` 继续表示业务人员档案，并新增或明确以下字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 人员 ID，沿用现有值 |
| `name` | 姓名 |
| `phone` | 唯一手机号 |
| `group_id` | 当前用户分组 ID |
| `group_name_snapshot` | 分组名称快照 |
| `group_locked` | 后台是否锁定分组 |
| `enabled` | 是否启用 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

现有 `role` 字段在迁移期保留用于兼容读取，但不再作为授权依据。授权只读取 RBAC 关系。

### 账户表 `accounts`

账户与人员一对一关联：

| 字段 | 说明 |
| --- | --- |
| `id` | 账户 ID |
| `person_id` | 唯一人员 ID |
| `login_name` | 当前等于规范化手机号，唯一 |
| `enabled` | 账户是否允许登录 |
| `auth_version` | 鉴权版本，用于批量撤销旧会话 |
| `last_login_at` | 最后登录时间 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

人员或账户任一处停用都视为不可登录。手机号修改时同步更新 `login_name`。

### 凭据表 `account_credentials`

只为需要后台密码登录的账户保存凭据：

| 字段 | 说明 |
| --- | --- |
| `account_id` | 账户 ID |
| `password_hash` | 使用成熟密码哈希算法生成的摘要 |
| `password_changed_at` | 最近修改时间 |
| `must_change_password` | 是否要求首次登录后修改密码 |
| `failed_attempts` | 连续失败次数 |
| `locked_until` | 临时锁定截止时间 |

不保存明文密码。设置、重置或修改密码时递增账户 `auth_version`，使旧后台会话失效。

### RBAC 表

`roles`：

- 每个用户分组对应一个系统管理角色。
- `source_group_id` 与用户分组一对一。
- 角色名称跟随分组名称更新。
- 角色停用状态跟随分组启用状态。

`account_roles`：

- 保存账户与角色的关联。
- 每个账户第一版只能有一个由所属分组同步而来的角色。
- 新建账户、修改分组或迁移旧用户时同步更新该关联。
- 后台不提供独立分配角色的入口，避免绕过“权限只由分组继承”的已确认规则。

`permissions`：

- 预置四项固定权限。
- 权限编码稳定，不跟随界面文案变化。

`role_permissions`：

- 保存角色与权限的多对多关系。
- 用户权限通过 `accounts -> account_roles -> roles -> role_permissions` 计算。
- 服务端同时校验角色的 `source_group_id` 与人员当前 `group_id` 一致；不一致时拒绝授权并触发关联修复。

该结构保留标准 RBAC 的账户、角色、权限三层关系，同时把角色分配约束为系统根据用户分组同步，不允许用户级权限覆盖。

### 会话表 `account_sessions`

| 字段 | 说明 |
| --- | --- |
| `id` | 会话 ID |
| `account_id` | 账户 ID |
| `session_type` | `mobile` 或 `admin` |
| `token_hash` | 随机会话令牌的哈希 |
| `auth_version` | 创建会话时的账户鉴权版本 |
| `expires_at` | 过期时间 |
| `last_seen_at` | 最近使用时间 |
| `revoked_at` | 撤销时间 |
| `created_at` | 创建时间 |

浏览器通过 `HttpOnly`、`SameSite=Lax` Cookie 持有随机令牌。服务端只保存令牌哈希。管理员后台会话与移动端会话分开，后台接口只接受 `admin` 会话。

### 微信与企微身份约束

在现有 `chat_identities` 基础上增加约束：

- 保留 `(platform, external_user_id)` 唯一约束。
- 增加同一人员、同一平台最多一条正式绑定的唯一约束。
- 临时身份不能直接作为后台绑定目标，必须先获得稳定外部标识。
- 换绑在事务中解除旧关系并建立新关系。

### 导入任务

复用 `import_jobs` 和 `import_job_rows`，增加人员导入所需的结构化结果：

- 导入任务类型为 `people`。
- 预览保存源文件摘要、解析结果、冲突快照和预览版本。
- 每行保存规范化数据、校验错误、冲突类型和管理员选择的动作。
- 提交后保存实际结果与失败原因。

预览数据不得包含后台密码。

## 后台信息架构

左侧一级导航新增“用户与权限”，与“工单设置”“展览数据”“系统配置”并列。

### 用户列表

主视图使用紧凑表格，支持：

- 按姓名、手机号、微信标识或企微标识搜索。
- 按分组、启用状态、管理员权限和绑定状态筛选。
- 分页或服务端限制返回数量，避免用户量增长后一次加载全部记录。

表格列：

- 姓名
- 手机号
- 分组
- 权限摘要
- 微信
- 企微
- 状态
- 最后登录
- 操作

主要操作包括新增、编辑、停用或启用、绑定管理、密码设置或重置、删除和批量导入。

### 用户编辑

编辑抽屉或独立详情区域包含：

- 姓名
- 手机号
- 所属分组
- 分组锁定
- 启用状态
- 当前继承权限，只读展示
- 微信绑定
- 企微绑定
- 后台密码状态
- 最近登录时间和最近变更摘要

当用户不具备 `admin.access` 时，不显示设置后台密码的主操作。已有凭据可以保留，但后台登录仍必须同时满足当前权限和启用状态。

“可用后台管理员”明确指同时满足以下条件的账户：

- 人员和账户均启用。
- 当前角色拥有 `admin.access`。
- 已设置有效后台密码凭据。

### 分组权限

现有“用户分组”配置继续维护名称、说明、认领、处理、验收、后台管理和启用状态。保存时同时更新：

- `user_groups`
- 对应 `roles`
- 对应 `role_permissions`
- 属于该分组的 `account_roles`

任何可能导致最后一位后台管理员失去权限的保存操作必须被服务端拒绝。

### 首位管理员引导

系统没有任何可用后台管理员时，`/admin` 显示一次性初始化入口：

1. 使用现有后台口令验证引导权限。
2. 填写首位管理员的姓名、手机号和后台密码。
3. 选择或创建具备后台管理权限的启用分组。
4. 创建人员、账户、凭据和后台会话。
5. 写入初始化完成标记。

初始化完成后，旧统一口令入口永久关闭。后续即使所有管理员被错误停用，也不自动重新开放旧口令；应通过受控运维恢复，避免形成长期后门。

## API 与服务边界

### 认证接口

- `POST /api/auth/mobile/login`
- `POST /api/auth/mobile/logout`
- `GET /api/auth/session`
- `POST /api/admin/auth/login`
- `POST /api/admin/auth/logout`
- `POST /api/admin/auth/bootstrap`

移动端登录返回当前服务端人员资料和权限摘要。后台登录必须验证手机号、密码、启用状态和 `admin.access`。

### 用户管理接口

- `GET /api/admin/users`
- `POST /api/admin/users`
- `GET /api/admin/users/{id}`
- `PATCH /api/admin/users/{id}`
- `DELETE /api/admin/users/{id}`
- `POST /api/admin/users/{id}/disable`
- `POST /api/admin/users/{id}/enable`
- `POST /api/admin/users/{id}/password`

所有接口从后台会话读取操作者，不接受客户端传入的管理员身份。

### 身份绑定接口

- `GET /api/admin/chat-identities`
- `PUT /api/admin/users/{id}/chat-identities/{platform}`
- `DELETE /api/admin/users/{id}/chat-identities/{platform}`

绑定请求可携带已有身份 ID，或稳定的 `externalUserId` 与显示名。冲突响应返回当前绑定用户摘要和确认令牌；只有携带该确认令牌的第二次请求才能换绑。

### 导入接口

- `POST /api/admin/user-imports/preview`
- `PATCH /api/admin/user-imports/{jobId}/rows`
- `POST /api/admin/user-imports/{jobId}/commit`
- `GET /api/admin/user-imports/{jobId}`
- `GET /api/admin/user-imports/{jobId}/report`

预览和提交分离。提交操作使用事务，并在落库前重新检查手机号、分组和身份占用状态。

### 工单接口鉴权

现有工单新建、认领、处理、验收、回复和催单接口改为：

- 从移动端或后台会话解析当前账户。
- 服务端读取当前人员、分组角色和权限。
- 忽略客户端传入的权限对象。
- 对需要操作者 ID、姓名和手机号的字段使用服务端会话快照。

查询权限维持现有产品行为；本次只强制四项已确认的操作权限。

## 批量导入

### 模板字段

支持 `.xlsx`、`.xls` 和 `.csv`。标准列为：

- `姓名`
- `手机号`
- `分组`
- `分组锁定`
- `启用状态`
- `微信账号标识`
- `企微账号标识`

布尔字段接受受控值，例如“是/否”“启用/停用”“true/false”，解析后统一为布尔值。姓名、手机号和分组为必填字段；微信和企微标识可空。

### 预览分类

每一行进入以下一种主状态：

- 可新增
- 手机号已存在
- 微信身份冲突
- 企微身份冲突
- 分组不存在或已停用
- 字段格式错误
- 文件内重复

同一行可以携带多个冲突明细。格式错误行不能提交。

### 管理员决策

管理员可以逐行或批量选择：

- 新增
- 覆盖已有用户
- 跳过

身份已绑定其他用户时，必须单独确认换绑。普通“覆盖”不能隐式抢占微信或企微身份。

对于手机号不存在的行只能选择新增或跳过；对于手机号已存在的行只能选择覆盖或跳过。

### 提交

提交时：

1. 校验导入任务属于当前管理员且尚未提交。
2. 校验预览版本和源文件摘要。
3. 重新读取用户、分组和身份当前状态。
4. 数据已变化的冲突行标记为失败，不使用过期决策覆盖新数据。
5. 在单个数据库事务中提交所有仍有效的选中行。
6. 写入导入行结果和审计日志。

为避免一行错误导致全部有效数据丢失，提交服务在事务内先完成全量验证；只有所有选中行均可按当前决策执行时才整体提交。若验证失败，任务保持可修改状态，管理员修正冲突后再次提交。

### 导入报告

导入结束展示：

- 总行数
- 新增数
- 覆盖数
- 跳过数
- 失败数
- 逐行结果和原因

报告可以下载为 `.xlsx`，包含原始行号、手机号、执行动作、结果和错误原因。

## 会话失效与安全

- 停用人员或账户时撤销其所有会话。
- 修改手机号时撤销其所有会话，要求重新登录。
- 修改分组、锁定状态或角色权限时递增受影响账户的 `auth_version`。
- 修改或重置密码时递增 `auth_version`。
- 后台接口要求有效 `admin` 会话和实时 `admin.access` 权限。
- 密码登录增加失败次数限制和短时锁定。
- 登录、退出、失败登录、密码变更、权限变更、换绑和导入均写入审计日志。
- Cookie 不暴露给客户端脚本，生产环境启用 `Secure`。
- 管理员不能查看已有密码，只能设置或重置。

## 文件存储兼容

项目仍支持 MariaDB 不可用时回退到 JSON 文件存储，因此仓储接口必须同时实现：

- 账户和凭据读写
- RBAC 角色与权限读写
- 会话创建、验证和撤销
- 用户查询和变更
- 身份绑定事务语义
- 导入预览与提交

JSON 模式下使用仓储级互斥锁和原子文件替换，避免并发登录、换绑或导入造成状态覆盖。密码哈希、令牌哈希和权限判断规则与 MariaDB 模式保持一致。

## 迁移方案

### 数据库迁移

新增迁移文件，完成：

1. 为 `people` 增加 `group_locked`，补齐可靠的 `group_id`。
2. 创建 `accounts`、`account_credentials`、`roles`、`account_roles`、`permissions`、`role_permissions` 和 `account_sessions`。
3. 为微信/企微身份增加同一人员、同一平台最多一个绑定的约束。
4. 扩展导入任务所需字段或新增人员导入预览表。
5. 写入四项固定权限。
6. 为每个现有用户分组创建对应角色和权限映射。
7. 为每个现有人员创建账户和账户角色关联，规范化手机号作为登录名。

迁移保留现有人员 ID、微信身份 ID 和历史工单引用，不批量重写业务历史。

### 旧角色兼容

现有 `people.role` 根据分组权限生成，迁移后保留但仅作为兼容快照。新代码不得使用该字段做授权判断。

### 旧登录状态

原有 `localStorage` 移动端身份和后台统一口令会话不再有效。升级后：

- 移动端用户重新免密登录一次。
- 管理员通过一次性初始化创建首位管理员。
- 初始化完成后使用手机号和密码登录后台。

## 错误处理

- 手机号重复：返回已存在用户摘要，不自动覆盖。
- 手机号格式错误：登录、新建、编辑和导入均拒绝。
- 分组不存在或停用：拒绝设置；移动端登录要求重新选择启用分组。
- 锁定分组冲突：移动端登录成功，但返回后台锁定后的实际分组。
- 身份占用：返回冲突详情，必须确认换绑。
- 临时微信身份：不允许后台绑定，提示等待稳定标识。
- 删除存在历史关联的用户：返回“该用户已有历史记录，仅可停用”。
- 最后一位管理员保护：拒绝导致系统失去可用管理员的操作。
- 导入数据过期：阻止受影响行提交，要求重新预览或处理冲突。
- 会话失效：接口返回 `401`；权限不足返回 `403`。

## 验收标准

- 后台管理员可以查看和筛选所有已建档用户。
- 管理员可以修改姓名、手机号、分组、分组锁定和启用状态。
- 用户权限始终由当前分组继承。
- 移动端首次免密登录自动建档，后续登录按锁定规则同步资料。
- 停用用户无法继续使用现有会话或重新登录。
- 后台只允许拥有后台管理权限并通过密码验证的用户进入。
- 系统可以使用旧后台口令完成一次首位管理员初始化，之后不再接受旧口令。
- 每个用户最多绑定一个微信身份和一个企微身份。
- 身份冲突不会自动换绑，必须由管理员确认。
- 批量导入支持预览、逐条或批量决策、事务提交和结果下载。
- 无历史用户可删除；有历史用户只能停用。
- 所有敏感变更均可在审计日志中追溯到具体管理员。
