# PC 后台页面 UI 设计

## 背景

当前系统已经有独立的 `/admin` PC 后台入口，包含后台工作台、微信下单日志、工单设置、展览数据和系统配置。现有页面功能可用，但界面仍保留较多移动端和表单堆叠痕迹：顶部导航分散，首页状态判断不够集中，配置表单扫描成本偏高。

本次设计目标是把后台改造成更务实的 PC 管理界面。页面第一屏要帮助管理员快速判断系统运行状态、异常风险和待处理事项，同时保持清晰的模块入口，便于后续扩展配置、日志和数据管理能力。

## 目标

- 将 `/admin` 设计为综合工作台，而不是纯配置页。
- 使用稳定的左侧导航承载后台一级模块。
- 首页展示运营总控信息：未关闭工单、待处理工单、失败通知、自动建单或消息接入状态。
- 将微信下单动态、待处理与异常队列、快捷配置入口放在首页核心区域。
- 配置类页面保留编辑能力，但改成更适合 PC 的分区表单和表格化布局。
- 视觉上保持专业、克制、易扫描，避免营销式大视觉和装饰性卡片。

## 非目标

- 不在本次设计中重做后端 API 或数据模型。
- 不改变现有 `/admin` 登录边界。
- 不引入完整权限系统。
- 不重写移动端主应用的工单业务流。
- 不把所有配置表单塞回首页。

## 推荐方案

采用“左侧完整导航骨架 + 运营总控首页”的方案。

左侧导航固定展示后台一级模块：后台工作台、微信下单日志、工单设置、展览数据、系统配置。导航项使用图标、文字和少量状态计数，保持模块名完整，避免只用图标造成识别成本。

首页作为综合工作台，顶部显示页面标题、说明、全局搜索或筛选入口、刷新和主要操作。主体分为三层：关键指标卡、待处理与异常队列、右侧辅助信息区。关键指标卡展示当前运行状态；异常队列优先展示需要人工介入的事项；右侧展示微信下单动态和快捷配置入口。

这个方案吸收了运营塔台式首页的判断效率，同时保留完整侧栏导航的可维护性，适合多人长期使用的后台。

## 信息架构

一级模块：

- 后台工作台：默认首页，展示运营状态、异常、微信动态和快捷配置。
- 微信下单日志：查看微信/企微消息分析、自动建单、匹配已有工单和待人工确认记录。
- 工单设置：维护用户分组、问题类型、关键词规则和自动分派相关配置。
- 展览数据：展示展位主数据概况，承接上传导入、校验结果和数据质量提示。
- 系统配置：维护 AI 模型、微信/企微 MCP 接入、系统级开关和接口参数。

首页内容优先级：

- 第一优先级：风险和待办，包括待处理工单、失败通知、待人工确认消息。
- 第二优先级：系统健康，包括消息接入状态、AI 模型启用状态、自动建单率。
- 第三优先级：常用配置入口，包括用户分组、问题类型、关键词规则、AI 接口。

## 主要页面布局

### 后台工作台

使用两栏 PC 布局：

- 左侧：固定宽度导航，品牌区在顶部，模块导航在中部，系统在线或接入状态在底部。
- 右侧：主工作区，顶部工具栏固定在内容上方，主体内容可滚动。

主工作区第一屏：

- 指标卡：未关闭工单、待处理工单、失败通知、自动建单率或消息接入状态。
- 待处理与异常队列：使用表格化列表，字段包括事项、来源、状态、负责人、时间。
- 微信下单动态：使用时间线或紧凑列表，展示最近处理结果。
- 快捷配置：以小型入口卡展示用户分组、问题类型、关键词规则、AI 接口。

### 微信下单日志

使用筛选栏 + 日志表格：

- 筛选项包括渠道、动作、状态、时间范围和关键词。
- 表格字段包括时间、渠道、动作、摘要、关联工单、状态。
- 对“待人工确认”“通知失败”等状态使用语义标签，避免仅用颜色表达。

### 工单设置

拆成清晰配置区块：

- 用户分组：使用表格或紧凑编辑行展示名称、说明、权限、启用状态和操作。
- 问题类型：使用表格展示名称、催单分钟、优先权重、默认处理组、启用状态。
- 关键词规则：使用分组卡 + 行内规则表，关键词输入保持可批量编辑。

每个区块顶部提供区块标题、简短说明和保存按钮，避免一个页面只有底部保存入口。

### 展览数据

优先展示数据资产状态：

- 当前展位数据总数、最近导入时间、导入状态、错误行数量。
- 上传入口和导入说明放在同一区块。
- 后续可以扩展数据预览表、校验结果和失败原因下载。

### 系统配置

使用分区表单：

- AI 接口：快速模型和高智能模型分开编辑，字段包括 provider、endpoint、modelName、apiKeyEnv、timeoutMs、enabled。
- 微信/企微 MCP：按渠道分组展示服务名称、接收地址、密钥环境变量、启用状态和自动建单状态。
- 系统状态：展示当前存储模式、消息接入状态和最近保存结果。

## 视觉系统

整体风格是“克制的运营后台”。

- 背景：浅灰绿或中性浅底，避免大面积渐变和装饰背景。
- 主色：沿用现有绿色作为品牌和主要操作色。
- 语义色：琥珀用于警告和待处理，红色用于失败和危险操作，蓝色用于信息状态。
- 圆角：后台容器、卡片和按钮以 8px 为主，降低移动端软卡片感。
- 阴影：只在主容器和弹层使用轻阴影，普通卡片以边框区分层级。
- 字体：使用现有中文无衬线字体栈，数据数字使用更高字重和稳定行高。
- 图标：使用 lucide-react，导航和按钮图标保持同一线性风格。

## 交互规则

- 左侧导航当前模块要有明显选中态，并支持状态计数。
- 工具栏中的刷新、退出、主要操作按钮要保持固定位置和稳定尺寸。
- 表格行 hover 时只改变背景或边框，不造成布局位移。
- 保存操作显示加载、成功和失败状态；失败信息靠近相关表单区块。
- 删除或危险操作使用确认提示，并与普通保存按钮视觉区分。
- 所有图标按钮需要有可访问名称。
- 键盘焦点需要清晰可见，输入框、按钮、导航链接都要保留 focus-visible 样式。

## 响应式策略

本次重点是 PC 版，但页面应在窄屏下保持可用：

- 1200px 以上：完整左侧导航 + 多列工作台。
- 900px 到 1199px：侧栏保持，主体内容从两列降为单列。
- 900px 以下：侧栏可改为顶部导航或折叠抽屉，表格转为卡片列表。

不为 PC 后台牺牲移动端主应用体验；后台窄屏适配只保证可访问和可操作。

## 数据与状态

设计复用现有 `AdminBackendShell` 加载的数据：

- `tickets` 用于工单指标和异常队列。
- `messageRecords` 和 `wechatOrderLogs` 用于微信下单动态和日志页。
- `outboundMessages` 用于失败通知指标。
- `config` 用于用户分组、问题类型、关键词、AI 模型和消息接入状态。
- `booths` 用于展览数据概览。

首页不需要新增 API 才能实现基础版本；可以在客户端基于 bootstrap 数据计算指标。后续如果数据量变大，再增加后台汇总接口。

## 测试建议

- `/admin` 登录后默认展示后台工作台。
- 左侧导航能跳转到日志、工单设置、展览数据和系统配置页面。
- 工作台指标能根据 bootstrap 数据正确计算。
- 待处理与异常队列优先展示待受理、待再次处理、失败通知和待人工确认记录。
- 微信下单动态在有日志和无日志时都显示合理状态。
- 配置页保存成功、失败和加载中状态清晰可见。
- 关键按钮、导航链接和表单控件可通过键盘访问。
- 在 1440px、1280px、1024px 和 900px 附近检查布局不重叠、不横向溢出。

## 实施边界

实施时优先重构组件结构和 CSS：

- `AdminBackendShell` 负责登录态、数据加载、页面框架和退出。
- `AdminConfigCenter` 可拆分为工作台、日志、工单设置、展览数据和系统配置子组件。
- 全局 CSS 中后台相关样式应与移动端样式分区，减少互相影响。
- 现有功能行为保持不变，先完成视觉结构和信息架构升级。

后续如果需要进一步增强，可以增加日志筛选、导入结果详情、配置区块内保存和系统健康检查。
